# Drama Pipeline 설명서

## 개요

`drama_pipeline.js`는 대본(각본)을 터미널에 붙여넣으면 **3개의 전문 AI 에이전트**가 분석한 뒤, **최종 결정자(Final Arbiter)**가 모든 분석을 종합하여 **컷 플랜(Cut Plan)**을 생성하는 Node.js 스크립트입니다.

## 아키텍처

```
사용자 대본 입력
       │
       ▼
┌──────────────────────────────────┐
│  Phase 2: 3개 에이전트 (순차 실행)  │
│  모델: Gemma 4 26B (Ollama 로컬)   │
│                                    │
│  ① 대본 분석가 (Script Analyst)     │
│  ② 연출 에이전트 (Director Agent)   │
│  ③ 촬영 에이전트 (Cinematography)   │
└──────────────┬───────────────────┘
               │ 3개 분석 결과
               ▼
┌──────────────────────────────────┐
│  Phase 3: 최종 결정자              │
│  모델: Gemini 2.5 Flash (클라우드) │
│                                    │
│  3개 에이전트 출력을 종합하여        │
│  최종 컷 플랜 생성                  │
└──────────────┬───────────────────┘
               │
               ▼
         최종 컷 플랜 출력
```

## 에이전트별 역할

### 에이전트 1: 대본 분석가 (Script Analyst)

- **역할**: 시니어 스크립트 슈퍼바이저 겸 조감독
- **온도**: 0.3 (낮음 — 데이터 정확도 우선)
- **하는 일**:
  - 슬러그라인(INT/EXT, 장소, 시간) 파싱
  - 캐스트, 소품, 의상, 세트 드레싱 등 22개 산업 표준 카테고리 분류
  - 프렌치 씬(캐릭터 입퇴장) 경계 식별
  - K-드라마 구조의 "감정 엔진" (정서적 인터루드) 식별
  - 대본에 명시되지 않았지만 액션에 필요한 "숨겨진 요소" 추출
- **출력 형식**: YAML (장면 메타데이터 + 프로덕션 요소 목록)

### 에이전트 2: 연출 에이전트 (Director Agent)

- **역할**: 수석 영화 감독 겸 내러티브 아키텍트
- **온도**: 0.8 (높음 — 창의적 비트 설계)
- **하는 일**:
  - 장면을 "드라마틱 비트" 단위로 분할
  - 매멧(Mamet)의 3가지 질문 적용: "누가 무엇을 원하는가?", "못 얻으면 어떻게 되는가?", "왜 지금인가?"
  - 숏폼 플랫폼용 "엄지 멈춤(Thumb-Stop)" 후크 설계
  - K-드라마 특유의 슬로우번 텐션, 사회적 마찰 비트 배치
  - 비트 유형: Hook → Rising Tension → Tactic Shift → Affective Interlude → Climax → Out
- **출력 형식**: YAML (비트 아키텍처 — 목표, 갈등, 비트별 전술/액션/강도)

### 에이전트 3: 촬영 에이전트 (Cinematography Agent)

- **역할**: 마스터 시네마토그래퍼 겸 광학 물리학자
- **온도**: 0.5 (중간 — 기술적 정밀도 + 유연성)
- **하는 일**:
  - 드라마틱 비트를 구체적인 샷 리스트로 변환
  - 렌즈 심리학 적용 (14mm 불안감 ~ 85mm 친밀감)
  - 노출 삼각형(조리개, 셔터 스피드, ISO) 계산
  - 9:16 세로 포맷 Safe Zone 준수 (상단 20~70% 내 주요 액션 배치)
  - 조명 설계: Low Key(서스펜스), High Key(코미디), Side Light(감정 표현)
  - 카메라 움직임의 감정적 동기 부여
- **출력 형식**: YAML (샷 리스트 — 렌즈, 조리개, 피사계심도, 조명, 카메라 움직임)

### 최종 결정자 (Final Arbiter)

- **역할**: 총괄 프로듀서
- **모델**: Gemini 2.5 Flash (Google 클라우드 API)
- **하는 일**:
  - 3개 에이전트 분석의 교차 검증 (존재하지 않는 소품/캐릭터 참조 확인)
  - 광학적 타당성 검증 (물리적으로 불가능한 카메라 설정 플래그)
  - 연속성 확인 (의상 상태, 소품 사용, 조명 일관성)
  - 페이싱 종합 (비트 아키텍처 + 샷 리스트 → 통합 타임라인)
  - 9:16 Safe Zone 최종 확인
- **출력**: 검증 보고서 + 통합 컷 플랜 + 프로덕션 노트

## 에이전트 온도 설정 근거

| 에이전트 | 온도 | 이유 |
|---------|------|------|
| 대본 분석가 | 0.3 | 데이터 추출 정확도 — 할루시네이션 방지 |
| 연출 에이전트 | 0.8 | 창의적 비트 설계 — 반복 정체 방지 |
| 촬영 에이전트 | 0.5 | 기술적 정밀도 + 약간의 창의적 유연성 |

이 설정은 Gemma 4 MoE 아키텍처의 특성에 맞춘 것으로, 낮은 온도는 결정적 샘플링에 가깝고 높은 온도는 반복 붕괴(repetition collapse)를 방지합니다.

## 기술적 특징

### Thought Stripping (사고 제거)

Gemma 4의 `<|think|>` 토큰으로 생성된 내부 추론 블록을 최종 결정자에게 전달하기 전에 제거합니다. 이전 에이전트의 내부 독백이 다음 에이전트의 "지시 희석(instruction dilution)"을 유발하는 것을 방지합니다.

### YAML 출력 형식

에이전트 간 데이터 전달에 YAML을 사용합니다. 연구에 따르면 소형 언어 모델에서 YAML이 JSON(43-50%)이나 XML(34-44%)보다 높은 정확도(52-62%)를 보입니다.

### 순차 실행 모드

`CONFIG.sequentialAgents: true` — 단일 GPU 환경에서 3개 에이전트가 GPU를 놓고 경쟁하지 않도록 순차적으로 실행합니다. 멀티 GPU 환경에서는 `false`로 변경하여 병렬 실행이 가능합니다.

### 재시도 로직

Gemini API가 503(과부하) 또는 429(속도 제한) 오류를 반환하면 최대 3회까지 자동 재시도합니다 (5초, 10초 간격).

## 실행 방법

```powershell
# 1. Ollama가 실행 중인지 확인
ollama serve

# 2. Gemma 4 모델이 설치되어 있는지 확인
ollama pull gemma4:26b

# 3. .env 파일에 Gemini API 키가 설정되어 있는지 확인
# GEMINI_API_KEY=your-key-here

# 4. 스크립트 실행
node drama_pipeline.js

# 5. 대본을 붙여넣고 ---END--- 를 입력한 뒤 Enter
```

## 파일 구조

```
drama_pipeline.js
├── CONFIG                    # Ollama/Gemini 연결 설정
├── AGENT_TEMPERATURES        # 에이전트별 온도 설정
├── AGENT_PROMPTS             # 4개 에이전트 시스템 프롬프트 (지침)
├── callOllama()              # Ollama API 호출 (로컬)
├── callGemini()              # Gemini API 호출 (클라우드, 재시도 포함)
├── runAgent()                # 에이전트 실행 래퍼 (로깅, 스피너, 에러 처리)
├── stripThinkingTokens()     # Thought Stripping 유틸리티
├── readScript()              # 터미널에서 대본 입력 받기
└── main()                    # 메인 파이프라인 (Phase 1~4)
```
